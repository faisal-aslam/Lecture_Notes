This video gives you a bird's eye view of data compression algorithms and techniques. We won't go deep into any single algorithm here, but this high level view is critical for understanding the overall field of data compression and for seeing where everything we cover in this course actually fits. Throughout the video you'll see a series of tree diagrams showing the hierarchy of compression algorithms and techniques, and here is the first one.

As this diagram shows, every data compression algorithm falls into one of two broad categories, lossless and lossy. Before we go any further into the hierarchy, it's worth pausing to understand exactly what separates these two categories, so let's look at a simple block diagram.

Imagine you have some original data, which we'll call x. This data goes into a compressor, also called an encoder, and what comes out the other side is a smaller, compressed version of that data, which we'll call c. Later, when you want your data back, you take that compressed data c and feed it into a decompressor, also called a decoder, and what comes out is the recovered data, which we'll call x bar.

The entire difference between lossless and lossy compression comes down to one question, how does x bar compare to the original x. In lossless compression, x bar is always identical to x. No matter what the input was, you are guaranteed to get back the exact original data, bit for bit. In lossy compression, x bar is not identical to x. Some information was discarded during the encoding process and is gone for good, so what you get back is only an approximation of the original, not an exact copy.

It makes sense to use lossy compression when the data is multimedia, that is audio, video, or images. Losing some data in exchange for better compression might not even be visible to the naked eye or audible to human ears. In every other case, when you need the data to come back exactly as it was, such as the textual records of government employees, you resort to lossless compression instead.

Before we split lossless compression into its categories, it's worth introducing one idea that sits underneath this entire field, and that is the Shannon limit. Claude Shannon showed that for any source of data, there is a theoretical minimum number of bits per symbol that you need, on average, to represent that data without losing anything, and this minimum is called the entropy of the source. No lossless compression algorithm, no matter how clever, can ever beat this limit on average. So really, every single lossless compression algorithm we are about to discuss, whether it is decades old or brand new, is ultimately trying to do the same thing, and that is to get as close as possible to this Shannon limit, while still being practical enough in terms of speed and memory to actually use. Keep this idea in the back of your mind as we go through the rest of this video, because it is the thread that ties every algorithm together.

With that in mind, lossless compression can be further divided into four different categories. Number one, entropy coding. Number two, dictionary or LZ family. Number three, transform based compression. And number four, context modeling.

Let us first briefly describe each of these categories, and then we will see what resides inside each of them by zooming in and creating more tree diagrams.

Let's start by understanding entropy coding. We have separate videos dedicated to explaining entropy, entropy rate, and the Shannon source coding theorem in detail, but here we will restrict ourselves to a simple version. Entropy coding refers to any encoding scheme where less probable symbols or blocks get longer codes, and more probable symbols or blocks get shorter codes. This is exactly the idea that lets us approach the Shannon limit we just talked about, since spending fewer bits on the symbols that occur more often is precisely how you bring your average bits per symbol down toward the entropy of the source.

If we zoom into entropy coding, we find two further groups of algorithms, prefix coding and non-prefix coding. In prefix coding we have algorithms like Huffman coding, Shannon coding, and Shannon-Fano coding. Among these, Huffman coding is both the fastest and the optimal prefix code for stateless data. Having said that, Huffman coding is not, overall, the best entropy coder available. This is because, according to the Kraft-McMillan inequality, which we have a separate video on, prefix codes are forced to round code lengths to whole numbers of bits. As a result, Huffman coding assigns an integer length codeword to each symbol, and this integer constraint limits how close it can actually get to the Shannon limit.
